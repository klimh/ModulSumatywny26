Wstępny projekt schematu bazy danych relacyjnej dla systemu RehabSense (np. PostgreSQL / MySQL)

Zaproponowany schemat bazy pokrywa wszystkie funkcjonalności wymienione w specyfikacji wymagań (role, parowanie pacjent-fizjo, plany ćwiczeń, czat, sesje z AI, passy).

=========================================
1. TABELE UŻYTKOWNIKÓW I RÓL
=========================================

Tabela: User (Użytkownicy - dane logowania i rola)
- id (PK)
- email (VARCHAR, UNIQUE)
- password_hash (VARCHAR)
- role (ENUM: 'PATIENT', 'PHYSIOTHERAPIST', 'ADMIN')
- created_at (TIMESTAMP)

Tabela: Patient (Szczegóły profilu pacjenta)
- user_id (PK, FK -> User.id)
- first_name (VARCHAR)
- last_name (VARCHAR)
- streak_count (INT) - (Aktualna "passa" regularności - WF7)
- last_activity_date (DATE) - (Do resetowania/naliczania passy)

Tabela: Physiotherapist (Szczegóły profilu fizjoterapeuty)
- user_id (PK, FK -> User.id)
- first_name (VARCHAR)
- last_name (VARCHAR)
- specialization (VARCHAR)
- is_verified (BOOLEAN) - (Weryfikacja przez admina - WF13)

Tabela: Patient_Physiotherapist (Relacja parowania - przypisanie pacjenta do fizjoterapeuty)
- id (PK)
- patient_id (FK -> Patient.user_id)
- physio_id (FK -> Physiotherapist.user_id)
- status (ENUM: 'PENDING', 'ACCEPTED', 'REJECTED')
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)

=========================================
2. TABELE KOMUNIKACJI (CZAT)
=========================================

Tabela: Message (Czat tekstowy - WF4)
- id (PK)
- sender_id (FK -> User.id)
- receiver_id (FK -> User.id)
- content (TEXT)
- sent_at (TIMESTAMP)
- is_read (BOOLEAN)

=========================================
3. TABELE PLANÓW REHABILITACYJNYCH
=========================================

Tabela: Exercise (Katalog wszystkich ćwiczeń/wzorców w systemie)
- id (PK)
- name (VARCHAR)
- description (TEXT)
- video_url (VARCHAR) - (Filmik instruktażowy)
- mediapipe_pattern_data (JSON/TEXT) - (Dane wzorcowe dla modelu AI)

Tabela: RehabPlan (Plan stworzony przez fizjoterapeutę dla pacjenta - WF9)
- id (PK)
- physio_id (FK -> Physiotherapist.user_id)
- patient_id (FK -> Patient.user_id)
- title (VARCHAR)
- description (TEXT)
- is_active (BOOLEAN)
- created_at (TIMESTAMP)

Tabela: Plan_Exercise (Lista ćwiczeń przypięta do konkretnego planu)
- plan_id (PK, FK -> RehabPlan.id)
- exercise_id (PK, FK -> Exercise.id)
- expected_reps (INT) - (Oczekiwana liczba powtórzeń)
- expected_sets (INT) - (Oczekiwana liczba serii)
- order_index (INT) - (Kolejność wykonywania)

=========================================
4. TABELE SESJI, POSTĘPÓW I AI
=========================================

Tabela: Session (Pojedyncza sesja ćwiczeniowa wykonana przez pacjenta - WF6)
- id (PK)
- patient_id (FK -> Patient.user_id)
- plan_id (FK -> RehabPlan.id)
- started_at (TIMESTAMP)
- ended_at (TIMESTAMP)
- overall_feeling (INT/VARCHAR) - (Opcjonalny feedback pacjenta po treningu: np. poziom bólu 1-10)

Tabela: ExerciseResult (Wyniki z analizy MediaPipe dla pojedynczego ćwiczenia w sesji - WF11, WF12)
- id (PK)
- session_id (FK -> Session.id)
- exercise_id (FK -> Exercise.id)
- reps_completed (INT)
- average_accuracy (DECIMAL/FLOAT) - (Średnia dokładność ruchu w %, wymagane min. 75%)
- ai_feedback_notes (JSON/TEXT) - (Szczegóły np. błędy zakresu ruchu zebrane przez AI)


=========================================
RELACJE W SKRÓCIE (Klucze obce)
=========================================
1. Każdy Patient ma jednego Usera (1:1).
2. Każdy Physiotherapist ma jednego Usera (1:1).
3. Fizjoterapeuta <-> Pacjent (Wiele-do-Wielu przez tabelę Patient_Physiotherapist).
4. Wiadomości łączą dwóch dowolnych Userów (Nadawca -> Odbiorca).
5. Fizjoterapeuta tworzy RehabPlan dla konkretnego Pacjenta (1:N).
6. RehabPlan składa się z wielu Exercise poprzez tabelę Plan_Exercise (N:M).
7. Pacjent wykonuje Session przypisaną do RehabPlan (1:N).
8. W ramach Session zapisywane są ExerciseResult dla poszczególnych ćwiczeń (1:N), zawierające dokładność zmierzoną przez model AI.
